1 Why Do We Need DynamoDB?

MIA runs on AWS Lambda — a serverless function that spins up, processes a request, and shuts down. Lambda has no memory between requests.

Problem 1: Lambda Forgets User asks "What is the batch yield?" then "Break that down by site." Lambda has no idea what "that" refers to — the first request is gone.
Problem 2: Duplicate Work 10 users ask the same question → Lambda runs the full AI pipeline 10 times. That's 10x the cost and latency for the same answer.
Solution: mia_chat_sessions Store conversation history persistently so the AI remembers context across messages.
Solution: mia_session_turns Cache results so repeated questions return instantly without re-running the pipeline.

3 Table 1: mia_session_turns

What Is This Table?

Think of it as a flight recorder for every AI interaction. Every time a user asks a question, one row is written with what they asked, which path the AI took, and the answer summary.

Why It Exists

PurposeWhat It Enables
Query CachingBefore running the AI pipeline, check: "Has this exact question been answered before?" If yes, return cached answer instantly.
Audit TrailFor compliance: "What did the AI tell User X at 3pm yesterday?"
Analytics"Which KPIs are most asked about?" "How often does the AI take the slow path?"

Key Design

mia_session_turns
Partition Key (pk): {tenant_id}#{session_id}
    Example: "default#sess-abc123"

Sort Key (sk): {timestamp_iso}#{request_id}
    Example: "2026-03-09T10:30:15Z#req-xyz789"

Why This Design?

  • Partition Key combines tenant_id (multi-tenancy — different teams/orgs) with session_id (groups all turns in one conversation). Query: "Give me everything in this session" → single fast DynamoDB call.
  • Sort Key uses timestamp + request_id. Timestamp ensures chronological ordering. Request ID prevents collisions at the same millisecond.

GSI: QueryCacheIndex

Global Secondary Index
Partition Key: query_hash (String)
Sort Key: created_at_utc (String)

Projected Attributes: narrative_summary, visualization_config,
    generated_sql, agent_logs, route_type, matched_kpi, is_valid
What this enables When a user asks a question, we hash it and query this GSI: "Has this exact question been answered before?" If yes, return the cached result without running any agents. The main table is keyed by session — this GSI re-indexes by question content.

4 Table 2: mia_chat_sessions

What Is This Table?

Think of it as ChatGPT's sidebar — a list of your past conversations, and clicking one shows the full message thread. It stores the complete conversation history so the AI can understand follow-up questions.

Why It Exists

PurposeWhat It Enables
Conversation ContextWhen a user asks "Break that down by site," the AI knows what "that" refers to from previous messages.
Session ResumptionUser closes browser, comes back tomorrow, continues the conversation.
User ExperienceShow past sessions in sidebar with titles like "Batch Yield Analysis."

Key Design

mia_chat_sessions
Partition Key (pk): {tenant_id}#{user_id}
    Example: "default#harish"

Sort Key (sk):
    SESSION#{session_id} ← for session metadata
    MSG#{session_id}#{timestamp}#{index} ← for individual messages

Why This Design?

  • Partition Key is tenant_id#user_id. "Show me all of Harish's sessions" is a single Query — no GSI needed.
  • Sort Key uses prefixes to store two types of items in the same partition: session metadata (SESSION#...) and individual messages (MSG#...). This is called vertical partitioning.

Why Individual Messages (Not a JSON Blob)?

DynamoDB has a 400KB item size limit Storing all messages in one JSON blob risks hitting this limit for long sessions.
ScenarioMsg SizeCountTotalStatus
Short KPI session~2 KB1020 KBSafe
Medium analysis~5 KB20100 KBSafe
Long power-user~8 KB40320 KBRisky
Heavy session~10 KB50500 KBFAILS

With individual items, each message is 2-10 KB — well under the limit. A session can have thousands of messages safely.

BenefitJSON BlobIndividual Items
Write a new messageRead blob → parse → append → serialize → writeSingle PutItem call
Read last 20 messagesRead entire blobLimit=20, ScanIndexForward=False
Concurrent writesRace condition riskNo conflicts (append-only)
400KB riskGrows with sessionEach item is 2-10 KB

Common Access Patterns

What You WantDynamoDB Query
List all of Harish's sessionspk = "default#harish", sk begins_with "SESSION#"
Get last 20 messages in a sessionpk = "default#harish", sk begins_with "MSG#sess-abc123", Limit=20, Descending
Get session metadata onlyGetItem pk = "default#harish", sk = "SESSION#sess-abc123"
Delete a session + all messagesQuery all matching items → BatchWriteItem delete

TTL Strategy

  • Active sessions: TTL = last_active + 30 days. Resets on every new message.
  • Archived sessions: TTL = archive date + 90 days.
  • All messages inherit the session's TTL — when a session expires, its messages expire too.

5 How the Two Tables Work Together

Here's what happens when a user sends "What is the batch yield for Site A?"

1 Hash the question → "a1b2c3d4..."
2 Check mia_session_turns GSI — Has this question been answered before?
CACHE HIT
Return cached answer
<100ms
CACHE MISS
Run full AI pipeline
5-15 sec
3 Save trace to mia_session_turns (for future cache hits + audit)
4 Save messages to mia_chat_sessions (user question + AI answer as 2 items)
5 Update session META — last_active, message_count, reset TTL

Bottom Line DynamoDB costs are negligible (~$0.01 per 1000 questions). The real savings come from caching: if 30% of questions are repeated, you save $3-15 per 1000 questions in AI (Bedrock) costs.

PK tenant#session

TTL 2-24 hours (route-based)
GSI QueryCacheIndex (by query_hash)
